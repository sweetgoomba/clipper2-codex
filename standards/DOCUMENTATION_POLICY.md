# Documentation Policy

작성일: 2026-04-29

Codex 작업은 설계와 구현 기록을 분리해서 남긴다.

## 1. 문서 위치

### `context/`

프로젝트 이해를 위한 배경 문서.

예:

- 레거시 히스토리
- 현재 구조 요약
- 제품/팀 아키텍처 방향

### `design/`

구현 전 설계 문서.

예:

- 여러 대안 비교
- 아키텍처 결정 초안
- 기능별 spec
- 개정 필요 표시가 붙은 과거 초안

### `implementation/`

실제 구현 진행 기록.

예:

- 현재 할 일
- 완료한 작업
- 검증 결과
- 다음 세션 인계

### `standards/`

반복적으로 지켜야 하는 원칙.

예:

- SOLID
- process boundary
- 문서화 규칙
- 테스트/검증 기준

## 2. 작업 전

구현 시작 전 반드시 확인한다.

1. `implementation/TASKS.md`
2. 관련 `design/*.md`
3. `standards/SOLID_AND_BOUNDARIES.md`
4. 각 repo `git status --short --branch`

## 3. 작업 중

작업 범위가 바뀌면 즉시 `TASKS.md`를 업데이트한다.

예:

- 새 Phase 추가
- 기존 Phase 보류
- 사용자의 우선순위 변경
- 충돌/권한/환경 문제 발견

## 4. 작업 후

구현이 끝나면 `WORKLOG.md`에 반드시 기록한다.

필수 항목:

- 변경한 repo
- 변경한 주요 파일/모듈
- 설계상 의미
- 검증 명령과 결과
- 남은 위험

## 5. 다음 세션 인계

작업이 중간에 끝나거나 구조 변경이 컸다면 `NEXT_SESSION_PROMPT.md`를 업데이트한다.

포함할 것:

- 현재 큰 방향
- 마지막 완료 작업
- 작업트리 상태
- 다음에 읽어야 할 문서
- 다음 첫 작업
- 주의해야 할 사용자 변경

주의:

- 특정 PC/특정 시점의 임시 로컬 변경 상태를 장기 인계 조건처럼 고정하지 않는다.
- repo 상태는 각 세션 시작 시 `git status --short --branch`로 다시 확인한다.
- 집/회사/다른 PC 간 환경 차이는 원칙이나 설계가 아니라 "현재 세션 관찰값"으로만 기록한다.

## 5.1 Session log 축약 원칙

`session-logs/YYYY-MM-DD.log`는 간결하게 쓴다.

포함한다:

- 사용자 요구의 핵심
- root cause 또는 결정
- 주요 변경 파일
- 검증 명령과 pass/fail
- 다음 작업

포함하지 않는다:

- 긴 명령 출력
- 반복 조사 과정
- diff 전문
- 이미 다른 문서에 있는 상세 설명

## 6. 설계 문서 개정

기존 설계가 틀렸다고 판단되면 삭제하지 않는다.

대신:

- 문서 상단에 "개정 필요"를 표시한다.
- 새 설계 문서를 추가한다.
- `README.md`에서 어떤 문서를 우선해야 하는지 명시한다.

이유:

- 결정이 바뀐 맥락이 중요하다.
- 이전 대안이 왜 폐기됐는지 나중에 다시 확인할 수 있어야 한다.
